## わかったこと・気づき

### 観測されている事実

| 指標 | 奥村版（Issue #3）| 決定木ガイド版（Issue #5 Phase 3）|
|---|---|---|
| バイナリ完全一致 | **31.61%（165/522）** | **0% (0/584)** |
| リエンコードサイズ | 概ね同等 | **2.5〜3.5 倍** |
| ピクセル損失 | あり | あり（68〜1246 px） |

### 構造的な疑念

1. **「2.5〜3.5 倍に膨らむ」は誤チューニングではなく、誤った候補を体系的に選んでいるシグナル**
   - 決定木は「マッチを使う」と判断しているが、そのマッチの選び方が悪く、結果的に Direct Pixel に倒した方がマシなケースで強引に短いマッチを選んでいる可能性

2. **「候補インデックス」を学習対象にした設計が脆い**
   - `leaf_choice_index` は候補配列の位置 N。配列生成順が学習時と推論時で違えば、N は別物を指す
   - オリジナルエンコーダの真の選択が `find_optimal_matches()` の候補配列に含まれているとは限らない（first-diff 分析で 17.2% は候補外）
   - 含まれていないラベルは「学習不能」だが、誤った位置に正解として記録されているおそれ

3. **特徴量 distance が学習されたルールに登場しない**
   - Phase 2 の features リストには distance が含まれるが、生成されたルール 100 個には現れない
   - Gini 不純度の改善に寄与しなかったということだが、これは「distance が無関係」なのではなく、**他の特徴量で代替されている**だけかもしれない

4. **0% 完全一致は構造的バグの可能性**
   - 候補が 1 個しかない局面なら、ランダム選択でも一致するはず
   - 開始バイトのズレ（パディング 0x10-0x11 の差）で全ペイロードが見かけ上ズレているだけの可能性が**未検証**

### 何が駄目だったか（このセッションの失敗）

- **Bug #1 修正だけで突破できると期待したのは甘かった**
- **デバッグツールの実装を進める前に、戦略を見直すべきだった**（kako-jun の Opus 切り替えで気づいた）
- **「ペイロードがそもそも近いのか・遠いのか」の安いテストを後回しにした**
