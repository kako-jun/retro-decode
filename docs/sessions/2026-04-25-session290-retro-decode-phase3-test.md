## わかったこと・発見

### 1. パディング部分（0x10-0x11）の不一致
- オリジナルファイル：パディング = オリジナル値（例：`0xa8 0x00`）
- リエンコード版：パディング = ゼロフィル（`0x00 0x00`）
- **デコード側がこの値を読まない** → 実装上は問題なし
- ただし、完全バイナリ一致の障害になる

### 2. ペイロード（圧縮ピクセルデータ）の大幅な差異
- **決定木ルールが、オリジナルエンコーダの選択を再現していない**
- リエンコード版のサイズが著しく大きい（2.5～3.5倍）
- → マッチ候補の選択が根本的に異なっている

### 3. ピクセル損失がある理由
- 決定木ルール：別のマッチ候補を選択 → 異なるピクセルパターンを生成
- デコード時に異なるピクセルが再現される
- 圧縮なので可逆性は保証されない（意図した設計か否かは別）

## 原因の推定

### 高い可能性
1. **Phase 2（決定木学習）のラベルデータが間違っていた**
   - 「Leaf の選択」ラベル（候補インデックス）が誤記
   - 決定木が「間違った選択パターン」を学習

2. **Phase 3（Rust 実装）のルール翻訳が間違っていた**
   - Python の決定木を Rust に翻訳する際に誤り
   - 特に、浮動小数点比較（`<=` vs `<`）や候補インデックス計算

### 中程度の可能性
3. **特徴量の計算が異なっている**
   - リエンコード時の `image_x`, `image_y`, `ring_r`, `length` が、学習時と異なる
   - 結果として、決定木が異なるリーフに到達
